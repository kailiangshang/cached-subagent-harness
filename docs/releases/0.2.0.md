# cached-subagent-harness v0.2.0

Date: 2026-07-16

`v0.2.0` turns `harnessctl` into a normal cross-platform Release artifact.
The standalone Skill remains the product contract; the new binary distribution
removes the requirement to compile its deterministic runtime on every machine.

## Highlights

- Five native `harnessctl` archives built on matching GitHub-hosted runners.
- Deterministic archives containing only the executable and `LICENSE`.
- One exact `SHA256SUMS` manifest and fail-closed archive validation.
- Verified Bash installation for Linux, macOS, and WSL-style environments.
- Native PowerShell 7 installation for Windows x86-64.
- Exact-version download with an explicit locked Cargo fallback.
- Tag/version equality and complete-matrix publication gates.
- Commit-pinned GitHub Actions and Rust `1.96.1` release builds.

This distribution increment does not change routing, batching, Session reuse,
the Dashboard schema, or the 20 Skill invariants.

## Release Assets

- `harnessctl-v0.2.0-x86_64-unknown-linux-gnu.tar.gz`
- `harnessctl-v0.2.0-aarch64-unknown-linux-gnu.tar.gz`
- `harnessctl-v0.2.0-x86_64-apple-darwin.tar.gz`
- `harnessctl-v0.2.0-aarch64-apple-darwin.tar.gz`
- `harnessctl-v0.2.0-x86_64-pc-windows-msvc.zip`
- `SHA256SUMS`

Every platform archive contains `harnessctl` or `harnessctl.exe` plus
`LICENSE`. GitHub's generated source archives contain the repository and Skill.

## Install

Check out `v0.2.0`, then use the platform-native installer:

```bash
scripts/install.sh --binary-source auto
```

```powershell
pwsh -NoProfile -File scripts/install.ps1 -BinarySource Auto
```

The four acquisition modes are:

| Mode | Contract |
|---|---|
| `auto` | Verified exact-version download, then locked Cargo fallback |
| `download` | Verified download or nonzero failure |
| `build` | Locked Cargo build only |
| `none` | Skill only, with the missing runtime reported explicitly |

The Bash `--skip-build` flag remains a deprecated alias for `none`. Both
installers preserve the copied Skill if required runtime acquisition fails.

## Compatibility

Prebuilt targets are Linux x86-64/ARM64, macOS Intel/Apple Silicon, and Windows
x86-64. The installers write a Codex-compatible Skill layout. Claude Code,
OpenCode, and other agentic runtimes may use the same Skill when they expose an
equivalent Skill interface and their own discovery path; this Release does not
claim that every host has passed live certification.

Native compatibility certification is limited to the release runners:
Ubuntu 24.04 / glibc 2.39, macOS 15, and GitHub's current `windows-latest` x86-64
image. Older operating-system releases are not certified in `v0.2.0`; a target
triple is an artifact identity, not a universal minimum-OS promise. Use `build`
when the prebuilt runtime is incompatible with a target system.

## Token Evidence Boundary

This is a long-running Token-aware control plane, not a proven net-saving
benchmark result. The retained comparable large-batch Harness sample used
`5,053,165` effective Tokens against a fresh `2,642,029` Baseline, and the
earlier repeated-follow-up sample was also more expensive. `v0.2.0` therefore
prevents those known high-cost Session patterns with bounded micro-batches,
strict causal usage, and conservative reuse limits. It does not claim positive
end-to-end Token savings.

Offline prompt-shape checks remain regression gates only; they are not provider
billing telemetry.

## Security and Known Limitations

SHA-256 verifies that downloaded bytes match `SHA256SUMS`, and installers reject
unexpected archive members before replacing the runtime. The binaries are
unsigned: `v0.2.0` does not include Apple notarization, Windows Authenticode,
Linux package signing, an auto-updater, or package-manager distribution.

Tag and Release mutability is controlled by repository settings, not this
workflow. The publication process verifies the existing version tag and exact
asset set, but it does not claim that maintainers are technically unable to
replace them later.

Users who require a platform trust signature can select `build` and compile the
tagged source with Cargo's locked dependency graph.
